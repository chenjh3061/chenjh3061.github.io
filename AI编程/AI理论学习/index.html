发表于 2025-10-26 更新于 2026-04-19 分类于 技术 ， AI编程 阅读次数： 本文字数： 5.8k 阅读时长 ≈ 5 分钟

一、LLM 大语言模型

1.1 核心机制：自回归生成

大模型本质上是一台概率机器——每次推理，它做且只做一件事：预测下一个 Token 是什么，然后把刚生成的 Token 追加到上下文，循环往复，直到触发停止条件。这个过程叫做自回归生成（Autoregressive Generation）。

这一机制带来两个重要的工程含义：

  • 模型无法回头修改已生成的 Token，一旦写出就成为下一步的事实依据
  • 输出越长，累积误差越大，越长的推理链需要越高的模型能力

1.2 Token 切分：子词算法

模型并不以字符或完整单词为单位处理文本，而是使用子词切分算法（Subword Tokenization）。核心思想是：高频词保留为整体，低频词拆成更小的片段。

主流算法对比：

算法 全称 代表模型 特点
BPE Byte Pair Encoding GPT 系列、Claude 贪心合并高频词对，简洁高效
Unigram LM Unigram Language Model SentencePiece 概率模型，找最优子词划分
WordPiece WordPiece BERT、RoBERTa 与 BPE 类似，但用似然最大化合并

工程提醒：不同模型、不同版本的 Tokenizer 差异巨大。在做成本预算时，务必查阅当前模型版本的官方 Tokenizer 演示，切勿沿用旧模型经验。

1.3 Token 成本与调用策略

成本核算规律：

  • 长 Prompt + 短输出 = 更经济：输入 Token 通常比输出 Token 便宜，因为输入只需一次前向计算，而输出需要自回归逐步生成
  • RAG 场景要严控检索片段数量：每多召回一段文本，都会直接增加输入 Token，成本线性上升
  • 思维链模型（CoT/o 系列）的 Reasoning Token 按输出价格计费，成本显著高于普通调用，预算时要单独核算

降本实践清单：

  • 用 Prompt 压缩替代冗余描述（删去”你好，我想问你一个问题”类的客套话）
  • 多轮对话场景做滑动窗口裁剪，避免上下文无限膨胀
  • 简单分类任务优先用小模型，将大模型算力留给复杂推理

二、概率采样与解码参数

2.1 Token 生成的完整流程

模型每生成一个 Token，要经过三步：

Step 1 — 打分（Logits）

对词表中所有候选 Token 计算原始分数，分数越高代表模型认为该词越”合适”。

以补全”今天天气真__”为例：

候选 Token 原始分数（logit）
好 5.0
不错 3.2
棒 2.1
糟糕 0.5
紫色 -8.0

Step 2 — 变换为概率（Softmax）

原始分数不能直接当作概率，需经过 Softmax 归一化：

$$P(token_i) = \frac{e^{logit_i}}{\sum_j e^{logit_j}}$$

变换后：

候选 Token 概率
好 62%
不错 20%
棒 10%
糟糕 5%
紫色 ≈ 0%

Step 3 — 采样（Sampling）

模型按概率分布”抽签”，决定最终输出哪个 Token。

解码参数（Temperature、Top-p、Top-k、Penalty 等）就是在这三步中插手干预，控制生成质量和风格。

2.2 解码参数详解

Temperature（温度）

在 Softmax 之前，对所有 logit 除以温度值 T：

$$P(token_i) = \frac{e^{logit_i / T}}{\sum_j e^{logit_j / T}}$$

值域 效果 适用场景
T → 0 概率高度集中于最优 Token，近似贪心解码 代码生成、数学推理
T = 1 原始概率分布，不干预 通用问答
T > 1 概率分布趋于均匀，低概率词也可能被选中 创意写作、脑洞生成

典型推荐值：代码 0.0～0.2 / 问答 0.5～0.7 / 创意 0.8～1.2

Top-k（硬截断）

只保留概率最高的 k 个候选 Token，其余丢弃，再重新归一化后采样。

缺点：k 值固定，不同语境适应性差——分布平坦时 k=3 太保守，分布陡峭时 k=50 又太发散。

Top-p（核采样，Nucleus Sampling）

按概率从高到低累加，直到累积概率刚好超过阈值 p，取该最小集合作为候选池。

优势：动态调整候选数量，比 Top-k 更自适应。实践中 Top-p 优先于 Top-k，常用值 0.90 ～ 0.95。

Repetition Penalty（重复惩罚）

已出现过的 Token，在下次采样时其 logit 被除以 penalty 因子（> 1），防止”复读机”。

注意：值过大会导致模型刻意回避合理的复用词汇，影响连贯性。推荐范围：1.0 ～ 1.2。

Max Tokens / Stop Sequence（输出停止控制）

两者共同控制生成何时终止，机制不同，需配合使用：

参数 类型 行为 风险
Max Tokens 硬上限 达到上限强制截断，无论内容是否完整 JSON 缺括号、列表缺项、句子中断
Stop Sequences 软停止 模型生成到指定字符串时自动停止 设计不当会提前截断关键字段

工程建议：结构化输出（JSON、代码块）场景，必须把截断风险作为独立的失败路径处理，建议添加输出格式校验和重试逻辑。思维链模型输出量远大于普通模型，预算 Max Tokens 时要预留充足余量。

思维链推理模式特别说明：在 o1/o3/DeepSeek-R1 等推理模型的思考阶段，Temperature、Top-p、Penalty 参数均不生效——模型会以固定策略自由探索推理路径，以保证思考质量不被采样参数干扰。

参数速查表

2.3 Logprobs：模型置信度信号

部分 API（如 OpenAI）支持返回每个生成 Token 的对数概率（logprobs），可理解为模型对该 Token 的”确信程度”：

  • logprob 趋近于 0（如 -0.01）：模型非常确信
  • logprob 较小（如 -5.0）：模型”犹豫”，该位置存在不确定性

工程应用场景：

  • 置信度评估：提取结构化字段（如金额、日期）时，若对应 Token 的 logprob 偏低，可触发人工复核流程
  • 异常检测：监控生产环境中平均 logprob，若突然下降，说明可能出现 Prompt 漂移或输入数据分布偏移
  • 多候选排序：获取 Top-N 候选 Token 及其概率，用于构建纠错系统或二次打分模型

注意：开启 logprobs 会增加响应体积；并非所有模型供应商均支持，使用前请查阅目标 API 文档。

三、DeepSeek 技术架构

3.1 全局架构概览


DeepSeek 在架构层面的核心创新可归纳为三个方向，分别解决计算效率、显存瓶颈、推理速度三大问题：

技术 全称 解决的核心问题
MoE Mixture of Experts 计算量过大（训练/推理成本高）
MLA Multi-Head Latent Attention KV Cache 显存占用过大
MTP Multi-Token Prediction 自回归逐 Token 生成速度慢

3.2 MoE：混合专家模型

背景问题：Dense（稠密）模型每次推理都激活 100% 的参数，随着模型规模增大，计算成本呈线性甚至超线性增长。

MoE 核心思路：将模型中的 FFN（前馈网络）层替换为多个并行的”专家”网络，每次推理通过门控路由（Gating Network） 只激活其中 Top-k 个专家。


关键工程挑战——专家负载均衡：若不加干预，门控网络倾向于总选少数几个专家（”明星专家”），导致其他专家闲置。DeepSeek 引入辅助损失（Auxiliary Loss）来惩罚不均匀分布，强制专家被均衡调用。

实际收益：在总参数量不变的情况下，激活参数量只有 Dense 模型的约 1/8 ～ 1/4，训练和推理的计算量大幅下降，同等算力可支撑更大规模的模型。

3.3 MLA：多头潜在注意力机制

背景问题：标准 Multi-Head Attention（MHA）在推理时需要缓存所有历史 Token 的 Key 和 Value（即 KV Cache），显存占用随序列长度线性增长，严重限制了长上下文推理能力。

MLA 核心创新：对 Key 和 Value 做低秩压缩（Low-Rank Compression），只缓存压缩后的潜在向量，使用时再解压还原。

性能收益：KV Cache 体积压缩 50% 以上，长上下文场景（如 128K tokens）的显存压力显著降低，同等硬件条件下吞吐量可提升 20%～30%。

3.4 MTP：多 Token 预测

背景问题：标准自回归生成每次只预测一个 Token，解码步骤是瓶颈，无法并行化。

MTP 核心思路：训练时让模型同时预测未来多个 Token，推理时借助 Speculative Decoding（推测解码）批量验证，加速生成。



实际优势：

  • 推理吞吐量提升明显，尤其是批量推理和长文本生成场景
  • 训练目标更丰富（多步预测损失），模型对序列依赖的理解更深

四、Agent 智能体

4.1 什么是 Agent

AI Agent（智能体）是以 LLM 为核心大脑，能够代替用户自主完成复杂、多步骤任务的系统。与单次问答不同，Agent 强调自主性（Autonomy）和交互性（Interactivity）——它能主动拆解目标、调用工具、处理结果、再规划下一步。

Agent 的四要素：

$$\text{Agent} = \text{LLM} + \text{Planning} + \text{Memory} + \text{Tools}$$

要素 职责 实现方式
LLM 推理与决策核心 GPT-4o、Claude、DeepSeek-R1 等
Planning 拆分目标、制定步骤 ReAct、Plan-and-Execute、ToT
Memory 存储与检索历史信息 滑动窗口、向量数据库、KV 存储
Tools 执行外部动作 搜索、代码执行、API 调用、文件操作

4.2 Agent Loop 与核心挑战

Agent 的运行是一个循环（Agent Loop）：感知 → 推理 → 行动 → 观察 → 再推理，直到任务完成或触发中止条件。

最大工程挑战：上下文膨胀

每一轮 Action + Observation 都会追加到上下文中。随着循环推进，上下文不断增长，带来：

  • 推理质量下降（”Lost in the Middle”问题：模型对上下文中间部分的关注度显著低于头尾）
  • Token 成本激增
  • 超出上下文窗口导致截断或报错

缓解策略：滑动窗口裁剪、摘要压缩、外置长期记忆（向量数据库）、关键信息置顶。

4.3 Tools 调用标准

工具调用遵循两个层次的标准：

层次 标准 说明
底层数据格式 OpenAI Function Calling Schema 定义工具名称、参数类型、描述的 JSON 结构
应用通信接入 MCP（Model Context Protocol） Anthropic 提出的开放协议，解决多工具接入的标准化问题

MCP 的价值：在 MCP 出现之前，每个工具都需要开发者手动对接 JSON 字段，维护成本高。MCP 将工具的声明、调用、结果返回标准化，使 LLM 可以像”插拔插件”一样使用工具。

Skills（技能） 是更高阶的封装形态——将一组工具和对应的 Prompt 模板打包为可复用单元，进一步降低 Agent 开发门槛。


4.4 上下文工程（Context Engineering）

上下文工程是 Agent 系统中的核心工程能力，决定了”每次推理，LLM 能看到什么”：

层次 技术 说明
短期记忆 滑动窗口（Sliding Window） 只保留最近 N 轮对话，超出部分裁剪
长期记忆 向量数据库（Vector DB） 语义检索历史信息，按需注入上下文
外部知识 RAG（检索增强生成） 实时检索文档库，将相关片段注入 Prompt
上下文优化 摘要压缩 / 关键信息置顶 减少冗余、强化重要信号的位置

4.5 核心推理范式：ReAct

ReAct（Reasoning + Acting）是目前最主流的 Agent 推理范式，将思考（Thought）和行动（Action）交替进行：

优势：思考过程可读可调试；行动与推理紧耦合，降低幻觉风险。

局限：串行执行，无法并行调用多个工具；长链路下上下文压力大。

4.6 Agentic Workflows：智能体工作流

Agentic Workflows 是更宏观的工程方法论——将 LLM 嵌入一套有状态的工作流中，而不是作为单次调用的黑盒。

四大核心设计模式（Andrew Ng 提出）：

模式 核心思想 典型实现
Reflection（反思） 让模型检查、批评、修订自己的输出 自我评分 → 重写 → 再评分
Tool Use（工具使用） 为 LLM 配备外部执行能力 ReAct 中的 Acting 阶段
Planning（规划） 先制定多步计划，再逐步执行 Plan-and-Execute、ToT
Multi-Agent Collaboration（多智能体协作） 多个专职 Agent 分工合作 Orchestrator + Worker 架构

工程视角的核心价值：

Agentic Workflows 的本质是用后端工程思维重构 AI 能力——把”推理、记忆、反思、多实体协作”编排成可观测、可调试、可扩展的流水线。当前 AI 应用从”Demo 玩具”走向”工业级生产力”，这是最成熟的工程路径，而不必等待底层模型的下一次参数突破。

常见解码参数推荐值

场景 Temperature Top-p Repetition Penalty 备注
代码生成 0.0 ～ 0.2 0.95 1.05 ～ 1.1 追求确定性
精确问答 0.1 ～ 0.3 0.9 1.1 避免幻觉

常见问题排查

现象 可能原因 解决方向
输出内容重复 Temperature 过低 / Penalty 不足 提高 Temperature 至 0.7+，增大 Penalty
输出天马行空 Temperature 过高 降低至 0.3 ～ 0.5
